## 第 1 部分：使用视角——memsearch 能做什么？

### Slide 1：memsearch：让 Agent 真正拥有记忆

**开场语（照念版）**

各位好，今天这场分享，我们主要聊一个在日常用大模型、用 Claude Code 开发时，非常实用但又容易被忽视的能力：**记忆系统**。  
我们会以 `memsearch` 这个项目为主线，带大家从使用体验一路走到内部实现，最后看它怎么跟你们现有的 Agent、服务、工具链集成在一起。

**要点提示**

- `memsearch` 是一个专门服务于 Markdown 知识库的**语义记忆系统**。
- 它解决两类问题：
  - 团队到处都是 `.md` 文件：ADR、会议纪要、日报、开发笔记；
  - Agent 每次开新会话都像刚入职，完全不知道之前发生了什么。
- 今天的目标：
  - 从**用户视角**理解：怎么用它搜索、怎么给 Agent 加记忆；
  - 从**实现视角**理解：内部是怎么设计的，改代码要注意什么；
  - 从**集成与运维视角**理解：怎么落地到你们的真实项目里。

**过渡句**

我们先从最直观的地方开始：**为什么需要 memsearch，它帮我们解决哪些具体痛点。**

---

### Slide 2：为什么我们需要 memsearch

**讲解思路**

可以先抛两个具体场景：

1. 你和 Claude 连续几天迭代同一个功能。  
   - 周一：一起加了 Redis 缓存。  
   - 周三：你说“订单接口有点慢”，Claude 完全不知道两天前已经加过缓存，又从零开始分析。
2. 团队所有架构决策、Meeting Notes、Bug Insight 都写在 `.md` 里。  
   - 过几周再找一条信息，只能靠模糊印象加 `grep`，要么搜不到，要么结果一大堆。

**总结句（可照念）**

这些问题有一个共性：  

- LLM **天生失忆**，上下文窗口一关就什么都没了；  
- 传统方案要么是自己搭数据库 + 向量库，路由、索引、运维很重；  
- 要么是用黑盒 SaaS，数据不透明、迁移成本高，还有厂商锁定风险。  

`memsearch` 想做的，就是：**既让 Agent 拥有持续的「长期记忆」，又尽量不增加运维负担，而且保证数据是我们看得懂的 Markdown 文本。**

**过渡句**

那 memsearch 是怎么做到这一点的？它有两条非常核心的设计理念，我们先把这两条说清楚。

---

### Slide 3：核心理念一——Markdown 是唯一事实源

**讲解（照念 + 自由发挥）**

第一条核心设计哲学是：**Markdown 是唯一的事实源（Source of Truth）**。

- 所有知识、记忆，最终都以 `.md` 文件落在磁盘上：
  - `MEMORY.md` 放长期不变的事实和关键决策；
  - `memory/YYYY-MM-DD.md` 按天滚动记录日常事件、会话总结。
- 向量数据库 Milvus / Zilliz 在这里只是一个**可重建的索引层**：
  - 真正的知识在 Markdown；
  - Milvus 可以随时删掉，只要再跑一遍 `memsearch index` 就能完整恢复索引。

为什么要这样设计？

- **人类可读**：随时可以打开任何一条记忆，看今天聊了什么、做了什么决策。  
- **Git 友好**：`.md` 天然适合做版本管理，diff / blame / review 都非常清晰。  
- **零锁定**：哪天不用 memsearch 了，这些 Markdown 还在，可以轻松迁移到其它系统。

**讲师提示**

可以强调一句：  
「**不要把业务事实塞进 Milvus 字段里当真源**，任何长期有价值的信息，都应该写回 Markdown。」

**过渡句**

第二条理念是：在记忆布局和 Chunk ID 的设计上，memsearch 完全对齐了 OpenClaw，这对兼容和扩展非常关键。

---

### Slide 4：核心理念二——对齐 OpenClaw 记忆架构

**讲解**

`memsearch` 是独立项目，但在记忆结构上**刻意跟 OpenClaw 保持一致**：

- 布局上：
  - 一个全局 `MEMORY.md`；
  - 一组 `memory/YYYY-MM-DD.md` 的每日日志。
- Chunk ID 的构造：
  - `hash(markdown:source:startLine:endLine:contentHash:model)`
  - 里面包含：源文件路径、行号区间、内容哈希、使用的 embedding 模型名。

**好处讲清楚：**

- 可以**无缝复用**现有的 OpenClaw 记忆目录，不需要做迁移脚本；
- 可以非常精确地知道「哪一行内容」对应 Milvus 里的哪一条记录；
- 内容一改，哈希就变，只需对改动部分重新做 embedding，**成本非常可控**。

**过渡句**

有了这两条理念，我们再来看整个系统是怎么分层的，从上到下都有些什么组件。

### Slide 6：从 0 到 1——第一次搜索（CLI）

**讲解**

这一页只想证明一件事：**memsearch 的基本能力，只要几条命令就能跑起来。**

步骤非常简单：

1. 写两份 Markdown：
   - `MEMORY.md`：写团队成员、负责人、关键架构决策；
   - `memory/2026-02-10.md`：写当天的 standup 和决策。
2. 执行索引：
   - `memsearch index .`
   - 扫描当前目录所有 `.md`，切块、嵌入、写入 Milvus。
3. 执行搜索：
   - `memsearch search "Bob 最近做了什么？"`
   - 看到结果中包含 Source、Heading，以及前 500 字预览。

**现场 Demo 建议**

- 现场开一个空目录 `demo-memory`，把这几步敲出来；
- 特别演示：
  - 第一次 `memsearch index .` 后显示 `Indexed N chunks.`；
  - 再执行一次 `memsearch index .`，看到 `Indexed 0 chunks.`，强调：
    - **索引是增量和幂等的，不会重复浪费 embedding 调用。**

**过渡句**

### Slide 7：Python API——Recall / Think / Remember 三步循环

**讲解**

Python 里可以把 Agent 的主循环理解成三步：

- **Step 1：Recall —— 回忆**
  - `mem = MemSearch(paths=["./memory"])`
  - `await mem.index()` 做一次初始索引；
  - 每次有新提问：`memories = await mem.search(user_input, top_k=5)`。
- **Step 2：Think —— 思考**
  - 把 `memories` 的内容拼成一段上下文：
    - `context = "\n".join(f"- {m['content'][:200]}" for m in memories)`
  - 把这段 context 放进系统提示，带着记忆去调用 LLM。
- **Step 3：Remember —— 记住**
  - 把这次对话的关键信息写到当天的 `memory/YYYY-MM-DD.md`；
  - 再执行一次 `await mem.index()`，让新的记忆进入索引。

**Demo 建议**

- 可以直接展示 `docs/getting-started.md` 里的 OpenAI 示例：
  - 先写两条「Team / Decision」到 memory；
  - 执行 `agent_chat("谁是我们的 frontend lead？")`；
  - 再执行一次，展示它确实是“从记忆里”回答的。

**过渡句**

到这里，大家大概知道了「用起来什么感觉」。接下来我们看一个更高级的用法：**Claude Code 插件如何做到“装完就有记忆”。**

### Slide 8：Claude Code 插件——装完就有记忆

**讲解**

先对比「有插件」和「没插件」的两种世界：

- 没有插件：
  - 周一、周二、周三，你和 Claude 做了很多工作；
  - 每次 session 结束，上下文全部丢失；
  - 周五再问「订单缓存有问题」，Claude 完全不知道之前怎么实现的缓存。
- 有 memsearch 插件：
  - 每次 session 结束：
    - 插件会读取这次的 JSONL transcript；
    - 调用一个小模型（Haiku）总结要点；
    - 把总结写入 `.memsearch/memory/当天.md`；
  - 下次打开 Claude Code：
    - SessionStart hook 会加载最近几天的记忆；
    - 当问题需要历史上下文时，自动调用 `memory-recall` skill 进行检索。

**用户视角总结**

- 不需要学习新命令；
- 不需要思考「什么时候该保存记忆」；
- 所有记录与召回，插件都自动替你完成。

### Slide 10：核心模块划分——一张“地图”

**讲解**

给大家一张「源码地图」：

- `core.MemSearch`：
  - 高层入口类；
  - 绝大多数调用方只需要关心它。
- `store.MilvusStore`：
  - 所有跟 Milvus 的交互都在这里；
  - collection 的 schema、索引、混合检索、统计、删除全部集中在此。
- `chunker / scanner`：
  - Markdown → Chunk 的切片逻辑；
  - `max_chunk_size`、`overlap` 等参数都在这里生效。
- `config`：
  - 把默认值、全局配置、项目配置、CLI 参数合并为一个 `MemSearchConfig`。
- `compact + resilience`：
  - 统一管理 LLM 调用、重试、退避策略；
  - 避免因为一次网络抖动导致整个 compact 失败。
- `watcher`：
  - 利用 watchdog 实现目录级别的 Markdown 文件监听。
- `transcript`：
  - 针对 Claude JSONL transcript 的解析与格式化输出。

**过渡句**

有了这张地图，我们从最常用的入口 `MemSearch.index()` 入手，拆解一下「索引流程」到底做了什么。

---

### Slide 11：索引流程详细拆解

**讲解**

从 `MemSearch.index()` 出发，可以把内部逻辑拆成几步：

1. **扫描文件**
   - `scan_paths(self._paths)`：
     - 支持多个路径；
     - 支持文件和目录；
     - 可以选择忽略隐藏文件。
2. **逐文件索引**
   - 对每个 `ScannedFile` 调用 `_index_file(f)`：
     - 读取文件文本；
     - 用 `chunk_markdown` 切成若干 `Chunk`。
3. **计算 Chunk ID 与去重**
   - 每个 chunk 拥有：
     - `content_hash`：对内容做 SHA-256 截断；
     - `chunk_id`：`compute_chunk_id(source, start, end, content_hash, model)`；
     - 再加上 `user_id` 前缀用于用户隔离。
4. **删除 stale chunks**
   - 查询该 source 在 Milvus 中已有的所有 `chunk_hash`；
   - 与当前文件生成的 `chunk_id` 做差集，删除不再存在的那些。
5. **嵌入与写入**
   - 对需要新增的 chunk 文本调用 embedding provider；
   - 构造记录字典写入 `MilvusStore.upsert`。
6. **返回统计与日志**
   - 返回本次新增的 chunk 数；
   - 打上 `files / chunks / duration_ms` 的日志，方便之后排查问题。

**强调句**

要给晋级开发同学强调的一点是：**`index()` 是幂等且增量的**，在 CI 或定时任务里频繁调用也不会导致 embedding 成本爆炸。

**过渡句**

刚才多次提到 `Chunk`，它是整个系统的基本粒度，我们单独看一下它的设计。

### Slide 12：Chunk 与 Chunk ID 的设计

**讲解**

一个 `Chunk` 包含的信息非常丰富：

- `content`：片段文本；
- `source`：源文件绝对路径；
- `heading / heading_level`：最近的 Markdown 标题及层级；
- `start_line / end_line`：在源文件中的行号（1-based）；
- `content_hash`：对内容做 SHA-256 截断的 16 位哈希。

Chunk ID 的构造公式：

- `hash(markdown:source:startLine:endLine:contentHash:model)`

**每个字段的作用：**

- `source + 行号`：方便回溯与调试，知道某条结果来自哪一行；
- `content_hash`：保证内容变了，ID 一定变；
- `model`：切换 embedding 模型时不会和旧索引混在一起。

**总体好处**

- 不需要额外的 cache 表或 Redis：
  - 主键本身就是「是否已经嵌入过」的判断依据；
- 与 OpenClaw 完全兼容，可以共享记忆目录和 ID 规则。

**过渡句**

有了 Chunk，我们再看数据在 Milvus 里的存储结构，以及混合检索是如何实现的。

---

### Slide 13：存储层——MilvusStore

**讲解**

`MilvusStore` 是对 `pymilvus.MilvusClient` 的一层封装，但非常关键：

- 创建 collection 时会：
  
  - 定义字段：
    - `chunk_hash`：VARCHAR(64) 主键；
    - `embedding`：FLOAT_VECTOR(dim)；
    - `content`：VARCHAR(65535) 并启用 analyzer；
    - `sparse_vector`：SPARSE_FLOAT_VECTOR；
    - `source / heading / heading_level / start_line / end_line / user_id / memory_type`。
  - 定义 Function：
    - BM25 Function：输入 `content`，输出 `sparse_vector`；
  - 创建索引：
    - 在 `embedding` 上建 FLAT + COSINE 索引；
    - 在 `sparse_vector` 上建 SPARSE_INVERTED_INDEX + BM25 索引。

- 对外提供的主要方法：
  
  - `upsert(chunks)`：插入或更新 chunk；
  - `search(query_embedding, query_text, top_k, filter_expr, user_id)`：
    - 会自动合并 user_id 的过滤条件；
    - 使用 hybrid search + RRF 返回结果。
  - `query / delete_by_source / delete_by_hashes / count / drop` 等。

**提醒**

可以明确说一句：  
「**不要直接在业务代码里拿 MilvusClient 自己建表改 schema，一律通过 MilvusStore。**」

**过渡句**

接下来，我们细看一下混合检索的实现，以及为什么要联合 dense 和 BM25 两种信号。

---

### Slide 14：混合检索与 RRF

**讲解**

`memsearch` 的检索不是单一路径，而是**Hybrid Search + RRF**：

- Dense 路径：
  - 使用 embedding 向量，在 `embedding` 字段上做 COSINE 相似度搜索；
  - 擅长处理「语义相似」的问题。
- Sparse 路径（BM25）：
  - 使用 `query_text`，在 `sparse_vector` 字段上做 BM25 搜索；
  - 对关键字、错误码、配置项、函数名特别敏感。
- RRF（Reciprocal Rank Fusion）：
  - 对两路结果按排名进行融合；
  - 避免某一条路径的信号完全压制另一条。

**实现层面**

- 在 `MilvusStore.search` 里：
  - 构造两个 `AnnSearchRequest`：
    - 一个用于 embedding；
    - 一个用于 BM25。
  - 调用：
    - `self._client.hybrid_search(reqs=[dense_req, bm25_req], ranker=RRFRanker(k=60))`；
  - 得到融合后的 top_k 结果。

**过渡句**

索引和检索层的大体逻辑已经比较清楚了。接下来我们看一下配置系统如何支撑多环境和多 provider 的需求。

### Slide 15：配置系统与多环境支持

**讲解**

配置系统的目标是：**一套代码，在本地、测试、生产三种环境下都好用**。

- 配置结构 `MemSearchConfig`：
  
  - `milvus`：`uri / token / collection`
  - `embedding`：`provider / model`
  - `compact`：LLM provider/model + 超时/重试参数
  - `chunking`：`max_chunk_size / overlap_lines`
  - `watch`：`debounce_ms`
  - `memory`：`base_dir / user_id / short_memory_dir / long_memory_dir / keywords 等`
  - `rerank`：是否启用、provider、model、api_base、api_key_env 等

- 优先级链：
  
  1. 默认 dataclass 值；
  2. 全局 `~/.memsearch/config.toml`；
  3. 项目内 `.memsearch.toml`；
  4. CLI 参数（优先级最高）。

- 典型用法：
  
  - CLI 用 `_build_cli_overrides` 拼出覆盖；
  - 应用层可以直接 `resolve_config()` 获取最终配置。

**实战建议**

- 每个项目**固定维护**一个 `.memsearch.toml`，统一约定：
  - 这个项目用哪个 Milvus URI 和 collection；
  - 默认的 embedding provider 和 model 是什么；
  - memory 放在仓库的哪个目录。

**过渡句**

接下来我们再看一眼「调用外部 LLM / embedding API」时的韧性设计，compact 和 resilience 模块是怎么配合的。

---

### Slide 16：Resilience 与 Compact

**讲解**

当我们调用 OpenAI / Anthropic / Gemini 这类外部服务时，网络抖动、限流是常态。  
`memsearch` 用两部分来处理这类问题：

- `async_retry`：
  - 接收一个异步函数 `call()`；
  - 接收一个 `is_retryable(exc)` 的判断函数；
  - 配置最大重试次数、基础延迟和最大延迟；
  - 内部用指数退避重试，并统一记录日志。
- `is_retryable_external_exception`：
  - 根据 HTTP status code（429 / 5xx 等）和异常类名判断异常是否可重试。

而 `compact_chunks` 会：

- 把所有 chunk 的 content 聚合成一个大文本；
- 用内置或自定义 prompt 模板 `COMPACT_PROMPT` 或传入模板；
- 根据 `llm_provider` 选择：
  - OpenAI / Anthropic / Gemini 对应的 SDK 调用；
- 整个过程包裹在 `async_retry` 中，保证在临时错误下自动重试。

**讲师提示**

可以补一句：  
「如果以后要接入自建 LLM，只需要在 `compact.py` 新增一个 provider 分支，并复用 `async_retry` 即可。」

**过渡句**

到此为止，核心实现层的图景已经比较完整了。

---

## 第 3 部分：集成视角——与 Agent / 框架结合

### Slide 17：memsearch 与 Agent 框架的关系

**讲解**

这里要先把定位说清楚：

- `memsearch` **不是** 一个 Agent 框架：
  - 它不负责 Chain、Graph、Planner 这些高层逻辑；
- 它是一个**标准化的记忆后端**：
  - 提供「语义检索」（`search`）；
  - 提供「记忆写入」（`memory.*` + Markdown 文件）；
  - 提供「压缩与长期记忆」（`compact` / `memory.consolidate`）。

跟 LangChain / LangGraph / CrewAI 等框架的关系：

- 框架负责：
  - 什么时候要查记忆；
  - 查到的结果怎么和其他工具、模型调用组合在一起。
- memsearch 负责：
  - 一旦需要“问过去的自己”，就给出高质量、可回溯的记忆片段。

**过渡句**

具体到落地，我们先看大家用得比较多的 LangChain 是怎么集成 memsearch 的。

---

### Slide 18：与 LangChain 的集成模式

**讲解**

在 LangChain 中，最自然的方式是：实现一个 `BaseRetriever`。

核心代码思路：

- 在 `_get_relevant_documents` 中：
  - 使用 `asyncio.run(self.mem.search(query, top_k=self.top_k))` 调用 memsearch；
  - 把结果列表转为 `Document(page_content, metadata)` 列表。

使用方式：

- `mem = MemSearch(paths=["./memory/"])`，先 `await mem.index()`；
- `retriever = MemSearchRetriever(mem=mem, top_k=3)`；
- `docs = retriever.invoke("Redis caching")`；
- 再用 LCEL 把 `retriever | format_docs` 嵌入到 RAG pipeline 中。

**强调**

- 对 LangChain 来说，这只是一个普通的 retriever；
- 对我们来说，memsearch 保障了检索质量和可回溯性。

**过渡句**

同样的思想也完全适用于 LangGraph、CrewAI 这类框架，我们快速看一眼。

---

### Slide 19：LangGraph / CrewAI 等其它集成

**讲解**

模式非常统一：

- 定义一个工具函数，例如：
  
  ```python
  @tool
  def search_memory(query: str) -> str:
      """Search the knowledge base."""
      results = asyncio.run(mem.search(query, top_k=3))
      ...
  ```

- 然后把这个工具交给框架：
  
  - LangGraph：`create_react_agent(llm, [search_memory])`；
  - CrewAI：在 Agent 初始化时，把 `tools=[search_memory]` 传进去。

**总结**

- 业务框架决定：**什么时候** 调用 `search_memory`；
- 工具函数内部只做两件事：
  - 用 `mem.search` 查记忆；
  - 把结果整理成对 LLM 友好的格式（纯文本或结构化）。

**过渡句**

以上是「通用框架集成」的模式。  
接下来我们重点看一个深度集成的典型案例：**Claude Code 插件**。

---

### Slide 20：Claude Code 插件架构

**讲解**

Claude Code 插件可以看成是 memsearch 在 IDE 场景下的旗舰集成。  
它由三部分组成：

- **Hooks（ccplugin/hooks）**
  - `SessionStart`：会话开始时触发；
  - `UserPromptSubmit`：每次用户输入前触发；
  - `Stop`：每轮 assistant 回复后触发（异步）；
  - `SessionEnd`：会话结束时触发。
- **Skill（ccplugin/skills/memory-recall）**
  - 一个 `context: fork` 的子 Agent；
  - 专门负责记忆检索，用 `memsearch search / expand / transcript` 做三层访问。
- **memsearch CLI**
  - 所有 Hook 实际上都是在 shell 里调用 `memsearch` 命令；
  - 插件本身几乎不直接写 Python 逻辑，而是 orchestrate CLI。

**过渡句**

这套架构最重要的设计是「渐进式披露」（Progressive Disclosure）。我们用一页单独讲清楚。

---

### Slide 21：Progressive Disclosure 三层模型

**讲解**

当 Claude 判断需要“查一查历史记忆”时，它会触发 `memory-recall` skill，这个 skill 在 fork 出来的子 Agent 里，分三层执行：

- **L1：`memsearch search`**
  - 根据当前用户问题做一次语义检索；
  - 返回若干 chunk 片段，包含 source、heading、score 等；
  - 子 Agent 会先在内部对这些结果做初步筛选和摘要。
- **L2：`memsearch expand CHUNK_HASH`**
  - 对特别关键的 chunk，用 `expand` 把原 Markdown 的完整 section 拉出来；
  - 这样可以看到更完整的上下文，例如整个「缓存设计」小节。
- **L3：`memsearch transcript path.jsonl --turn UUID`**
  - 如果需要看到原始对话细节（包括工具调用、错误日志等），再通过 transcript 命令回放指定 turn 前后的若干轮对话。

**两个关键点：**

1. Skill 在**forked 子 Agent** 里跑，它的中间过程不会污染主对话上下文；
2. 主对话只收到一份最终的「记忆摘要」，而不是一大堆原始检索结果。

**过渡句**

Agent 侧的逻辑我们就说到这里。最后一部分，我们回到运维视角，看如何部署、配置、排障。

## 第 4 部分：运维视角——部署、配置与排障

### Slide 22：部署模式与环境规划

**讲解**

从运维角度看，memsearch 典型有三种部署模式：

- **本地开发：**
  
  - Milvus Lite：单个 `.db` 文件（例如 `~/.memsearch/milvus.db`）；
  - Embedding 可以选择 OpenAI 或本地 sentence-transformers；
  - 零运维，非常适合个人或小团队开发。

- **团队测试 / 内部环境：**
  
  - 部署一个 Milvus Server（Docker/K8s）；
  - 不同项目通过 Collection / user_id 做隔离；
  - Embedding provider 可以统一配置，方便管理成本。

- **生产环境：**
  
  - 使用 Zilliz Cloud 提供的托管 Milvus；
  - 可按业务维度规划多个 cluster / collection；
  - 项目内通过 `.memsearch.toml` 关联到对应的集群和 collection。

**过渡句**

部署完之后，下一步就是「如何合理管理配置」的问题。

---

### Slide 23：配置管理最佳实践

**讲解**

推荐团队采纳以下约定：

- 在每个项目根目录维护一个 `.memsearch.toml`，至少包含：
  - `milvus.uri / milvus.collection`：这个项目的索引存在哪里；
  - `embedding.provider / embedding.model`：统一项目使用的嵌入模型；
  - `memory.base_dir`：记忆文件相对于仓库的路径，比如 `memory` 或 `.memsearch/memory`。
- 所有 API Key 只通过环境变量提供：
  - `OPENAI_API_KEY / GOOGLE_API_KEY / VOYAGE_API_KEY / ANTHROPIC_API_KEY / OLLAMA_HOST` 等；
  - 不把密钥写入配置文件或代码仓库。
- 如需为某个用户/Agent 自定义 `user_id`：
  - 在 CLI 调用中加 `--user`，或者在构造 `MemSearch` 时传 `user_id`；
  - 统一在配置/调用层控制，避免硬编码散落在业务代码里。

**过渡句**

即便配置合理，还是难免会遇到各种问题。我们来看几个最常见的故障场景，以及推荐的排查路径。

---

### Slide 24：常见问题与排障流程

**讲解**

常见问题主要有三类：

1. **搜索没有结果**
   
   - 首先跑：`memsearch stats` 看当前 collection 里有多少 chunk；
   - 如果是 0：
     - 检查是否对正确的路径执行了 `memsearch index`；
   - 如果有 chunk：
     - 在 CLI 里试：`memsearch search "一段你确定存在的文本"`；
     - CLI 有结果但程序没有，多半是应用层集成问题；
     - 如果都没有结果，检查 `embedding.provider` 是否与索引时一致。

2. **配置不生效**
   
   - 使用 `memsearch config list --resolved` 查看最终配置；
   - 特别留意：
     - `milvus.uri / embedding.provider / embedding.model` 等关键字段；
   - 注意：CLI 参数优先级最高，可能覆盖 `.memsearch.toml`。

3. **Milvus 报 embedding 维度错误**
   
   - 常见原因：
     - 之前用某个 embedding 模型索引了 collection；
     - 后来切了模型，但还在用同一个 collection。
   - 解决方式：
     - 换一个新的 collection 名称；
     - 或者 `memsearch reset --yes` 后重新索引。

**过渡句**

此外，在 Claude Code 插件场景下，还有一套专门的排障方式，我们单独看一下。

---

### Slide 25：Claude Code 插件排障

**讲解**

在 Claude Code 中，memsearch 插件的状态可以从三处观察：

1. **SessionStart 状态行**
   
   - 每次开 session 时，终端上方会打印一行，比如：
     - `[memsearch v0.1.11] embedding: openai/text-embedding-3-small | milvus: ~/.memsearch/milvus.db`
   - 如果看到：
     - `ERROR: OPENAI_API_KEY not set` 之类的信息；
     - 说明当前 embedding provider 对应的 key 没配置好。

2. **Debug 日志**
   
   - 用 `claude --debug` 启动；
   - 查看 `~/.claude/logs/` 里的日志文件；
   - 搜索 hook 相关的 JSON 输出，重点看：
     - `systemMessage`；
     - `additionalContext` 是否包含最近几天的 memory 内容。

3. **Memory 文件**
   
   - 查看 `.memsearch/memory/YYYY-MM-DD.md`：
     - 正常情况下，每次 session 结束后会追加：
       - `## Session HH:MM` 标题；
       - 若干 bullet 的总结。
   - 如果只有 Session 标题，没有 bullet：
     - 多半是 `stop` hook 里调用 `claude -p` 失败；
     - 需要检查 CLAUDE CLI 是否可用、模型名和 key 是否正确。

### Slide 26：总结与下一步

**总结（照念版）**

最后做一个小结，今天主要想让大家记住三件事：

1. **架构理念**  
   
   - Markdown 是唯一事实源，Milvus 只是可重建的索引层；  
   - 整体设计与 OpenClaw 保持高度兼容，可以直接复用现有记忆资产。

2. **工程实现**  
   
   - 核心类 `MemSearch` 串联了 scanner → chunker → embeddings → MilvusStore 全链路；  
   - Hybrid Search + RRF 保障了检索质量；  
   - config / compact / resilience / watcher 等模块支撑多环境稳定运行。

3. **落地实践**  
   
   - CLI + Python API 让你们很容易把 memsearch 接入现有服务和 Agent；  
   - Claude Code 插件是一个端到端的真实示范；  
   - 与 LangChain / LangGraph / CrewAI 等框架的集成模式统一且简单。

**下一步建议**

- 选一个你们正在维护的项目，真实跑一遍：
  - 在仓库里加上 `memory` 目录；
  - 把关键 ADR、会议纪要迁移或复制过去；
  - 配好 `.memsearch.toml` 和 Claude 插件；
  - 让 Agent 在这个项目上「带着记忆」工作一段时间。
- 后续在接入或二次开发中遇到问题，可以根据今天的模块划分快速定位到对应源码。
