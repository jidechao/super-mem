    培训讲义

    memsearch 项目培训大纲¶

    本培训面向：后端工程师、工具链开发者、Agent 应用开发者。
    目标：理解 memsearch 的整体设计与源码结构，掌握在项目中正确使用和集成 memsearch 的方法。

    1. 使用视角：memsearch 能做什么¶

    1.1 典型痛点与问题背景¶

    • 语言模型“失忆”：对话结束上下文就消失，无法跨会话记忆。
    • 团队知识大量存在于 Markdown（ADR、会议纪要、日报、README），但检索体验差。
    • 传统方案多依赖数据库或黑盒 SaaS，存在：
    • 数据不可读，不好审计。
    • 运维成本高（备份、迁移、版本兼容）。
    • 厂商锁定风险。

    1.2 memsearch 的核心理念¶

    • Markdown 是唯一事实源（Source of Truth）
    • 知识以 .md 文件形式长期保存，人类可读、Git 友好。
    • Milvus / Zilliz 仅作为“可重建的索引层”，随时可丢弃并重建。
    • 完全对齐 OpenClaw 记忆架构
    • 文件布局：MEMORY.md + memory/YYYY-MM-DD.md。
    • Chunk ID 格式：hash(markdown:source:startLine:endLine:contentHash:model)。
    • 内容哈希去重：避免重复 embedding 调用。
    • 混合检索
    • Dense embedding + BM25 + RRF 融合，兼顾语义与精确关键字。

    1.3 第一次使用：从 0 到 1¶

    • 安装与依赖
    • pip install memsearch
    • 可选 extras：[google] [voyage] [ollama] [local] [anthropic] [all]
    • 准备一个内存目录：
    • MEMORY.md：长期事实和关键决策。
    • memory/YYYY-MM-DD.md：按天记录工作内容与会话总结。
    • 核心命令：
    • memsearch index PATHS：初次建索引。
    • memsearch search "your question"：语义检索。
    • memsearch search --json-output：作为工具链的一部分返回结构化 JSON。

    1.4 Python API 快速上手¶

    • 核心类：memsearch.MemSearch
    • 三步循环：
    • Recall：await mem.search(user_input, top_k=5)
    • Think：把检索到的记忆注入 LLM 上下文进行推理。
    • Remember：将本轮对话要点写入 Markdown，再 await mem.index()。

    1.5 Claude Code 插件（ccplugin）¶

    • 安装方式：Claude Code 插件市场 zilliztech/memsearch
    • 使用体验：
    • 无插件：新 session 完全不知道过去做过什么。
    • 有插件：自动总结历史 session 写入 .memsearch/memory/YYYY-MM-DD.md，在需要时自动召回。
    • 用户无须学习新指令，所有记忆相关操作通过 Hook + Skill 自动完成。

    2. 实现视角：核心架构与源码¶

    2.1 总体分层¶

    • 应用层
    • Claude Code 插件（Hooks + Skills）
    • 与 LangChain / LangGraph / LlamaIndex / CrewAI 的集成。
    • 服务层
    • CLI：memsearch 命令集合。
    • Python API：MemSearch 类。
    • 核心基础设施层
    • chunker / scanner / store / embeddings / compact / config / watcher / transcript / memory。

    2.2 索引流程¶

    • 步骤：
    • scan_paths(paths)：递归发现所有 Markdown 文件。
    • chunk_markdown(text, source, max_chunk_size, overlap_lines)：
      • 一级按标题分块，二级按段落拆分超长 section。
    • 计算每个 chunk 的 content_hash 和复合 chunk_id。
    • 查询该 source 旧有 chunk id 集合：
      • 删除已不在文件中的 stale chunks。
      • 对比决定哪些 chunk 需要重新嵌入。
    • 调用 embedding provider 批量嵌入，再通过 MilvusStore.upsert 写入 Milvus。
    • 特性：
    • 增量索引：重复 index() 基本不会产生成本。
    • 删除传播：删除 markdown 文件会自动清理对应向量数据。

    2.3 存储与混合检索¶

    • Milvus collection schema：
    • 主键：chunk_hash
    • 向量字段：embedding（dense）、sparse_vector（BM25）
    • 文本字段：content（用于 BM25）、source / heading / heading_level / start_line / end_line
    • 辅助字段：user_id / memory_type
    • Hybrid search 实现：
    • AnnSearchRequest 两路：embedding & BM25。
    • RRFRanker(k=60) 融合结果。
    • 用户隔离：
    • 通过 user_id 字段和 filter 表达式实现逻辑隔离。
    • 支持一个 Milvus 集群中多用户共用同一 collection。

    2.4 Markdown 处理与 Chunk 策略¶

    • Heading 级分块保证语义完整性。
    • Paragraph 级拆分控制块大小。
    • overlap_lines 保证上下文连续性。
    • content_hash + chunk_id 共同构成：
    • 去重键（避免重复 embedding）。
    • 可追踪键（定位具体来源文件与行号）。

    2.5 配置系统¶

    • 配置结构 MemSearchConfig：
    • milvus / embedding / compact / chunking / watch / memory / rerank
    • 优先级：
    • 默认值 → ~/.memsearch/config.toml → .memsearch.toml → CLI 覆盖。
    • 常见配置项：
    • Milvus：uri / token / collection
    • Embedding：provider / model
    • Memory：base_dir / user_id / 短长记忆目录 / 关键字触发器
    • Rerank：enabled / provider / model / api_base / api_key_env / timeout 等。

    2.6 Resilience 与 Compact¶

    • async_retry：统一的异步重试与指数退避。
    • compact_chunks：
    • 支持 OpenAI / Anthropic / Gemini。
    • 将多个 chunk 聚合成一个 markdown summary。
    • 自动写入 memory/YYYY-MM-DD.md 并立即 re-index。

    3. 集成视角：与 Agent / 框架的结合¶

    3.1 总体模式¶

    • 把 MemSearch 封装为“检索工具”，交由各框架调用：
    • LangChain：BaseRetriever 实现。
    • LangGraph：@tool 工具函数。
    • LlamaIndex：BaseRetriever 返回 NodeWithScore。
    • CrewAI：@tool("search_memory")。

    3.2 LangChain 示例要点¶

    • 用 asyncio.run(mem.search(...)) 在同步环境中调用。
    • 将每个结果转为 Document(page_content, metadata)。
    • 在 RAG chain 中组合：
    • {"context": retriever | format_docs, "question": RunnablePassthrough()} | prompt | llm | StrOutputParser()。

    3.3 Claude Code 插件详解¶

    • Hook 流程：
    • SessionStart：
      • 校验配置与 API Key。
      • 启动 memsearch watch，写 session heading。
      • 注入最近两天 memory 的尾部内容作为冷启动上下文。
    • UserPromptSubmit：
      • 简单提示 [memsearch] Memory available，提高 Skill 触发概率。
    • Stop（异步）：
      • 解析 transcript → 用 Haiku 总结 → 写入 .memsearch/memory/YYYY-MM-DD.md → 强制 memsearch index。
    • SessionEnd：
      • 清理 watch 进程与 PID。
    • Skill：memory-recall
    • 运行在 context: fork 子 Agent 中：
      • L1：memsearch search。
      • L2：memsearch expand。
      • L3：memsearch transcript。
    • 主对话只看到最终摘要，避免对话上下文被检索细节污染。

    4. 运维视角：部署、配置与排障¶

    4.1 部署模式¶

    • 本地开发：
    • Milvus Lite（~/.memsearch/milvus.db）+ OpenAI / local embedding。
    • 团队环境：
    • 独立 Milvus Server + 统一配置的 embedding.provider。
    • 生产环境：
    • Zilliz Cloud + 明确的 collection 规划 + 基于 user_id 的逻辑隔离。

    4.2 配置管理最佳实践¶

    • 建议每个项目维护 .memsearch.toml，固定：
    • Milvus 连接参数。
    • Embedding provider/model。
    • Memory 目录与 user_id 规则。
    • API Key 仅通过环境变量提供，不落盘。

    4.3 常见问题与排障清单¶

    • 搜索无结果：
    • memsearch stats 检查 chunk 数。
    • 确认 memsearch index 是否对正确路径运行。
    • 检查 embedding provider 是否与索引时一致。
    • 配置不生效：
    • 使用 memsearch config list --resolved 查看最终合并结果。
    • Milvus 报维度错误：
    • 说明更换了 embedding 模型；需要：
      • 使用新 collection，或
      • memsearch reset --yes + 重新 index。
    • Claude 插件问题：
    • 看 SessionStart 的 systemMessage 状态行。
    • 若出现 ERROR: XXX_API_KEY not set，检查环境变量。
    • 检查 .memsearch/memory/YYYY-MM-DD.md 是否持续产生内容。

    5. 总结与问答¶

    • 再次强调三点：
    • Markdown 是事实源，Milvus 是索引层。
    • MemSearch 提供稳定、可扩展的记忆基础设施。
    • Claude Code 插件是 memsearch 在真实场景中的标杆落地实现。
    • 建议后续实践：
    • 选一个正在进行的项目，落地 memsearch + Claude 插件。
    • 将架构 ADR / 重要会议纪要全部纳入 memory 体系中。
